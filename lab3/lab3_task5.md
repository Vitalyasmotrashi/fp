## Реализация 1: `lab3_task4_list` (на основе списка пар `[{Elem, Count}]`)

Внутренняя структура — связный список пар. Поиск, вставка и обновление требуют
линейного прохода по нему.

| Операция   | Сложность      | Причина |
|------------|----------------|---------|
| `new()`    | **O(1)**       | Возвращает константу `[]` |
| `add(E, MS)` | **O(n)**     | `lists:keyfind` — линейный поиск; `lists:keyreplace` — ещё один проход |
| `remove(E, MS)` | **O(n)** | Аналогично: keyfind + keydelete/keyreplace |
| `count(E, MS)`  | **O(n)** | `lists:keyfind` — линейный поиск |
| `member(E, MS)` | **O(n)** | Вызывает `count/2` |
| `to_list(MS)`   | **O(1)** | Возвращает саму структуру без преобразования |
| `size(MS)`      | **O(n)** | `lists:sum` + list comprehension — проход по всем парам |

**Итог для списка:** все операции, кроме `new` и `to_list`, — **O(n)**.

## Реализация 2: `lab3_task4_maps` (на основе `map`, Erlang OTP 17+)

Внутренняя структура — HAMT (Hash Array Mapped Trie) / сбалансированное дерево.
Erlang реализует maps через структуру данных, обеспечивающую **O(log n)** для
поиска и обновления.

| Операция   | Сложность        | Причина |
|------------|------------------|---------|
| `new()`    | **O(1)**         | Возвращает пустой map `#{}` |
| `add(E, MS)` | **O(log n)**   | `maps:update_with` — поиск и обновление в дереве |
| `remove(E, MS)` | **O(log n)**| `maps:get` + `maps:remove` / `maps:put` |
| `count(E, MS)`  | **O(log n)**| `maps:get` — поиск по ключу |
| `member(E, MS)` | **O(log n)**| Вызывает `count/2` |
| `to_list(MS)`   | **O(n log n)**| `maps:to_list` обходит все n ключей |
| `size(MS)`      | **O(n)**    | `maps:fold` — проход по всем парам |

**Итог для maps:** O(log n) для точечных операций, O(n) для bulk-операций.

---

## Сравнение двух реализаций

| Операция       | Список (list) | Map |
|----------------|:-------------:|:---:|
| `new`          | O(1) | O(1) |
| `add`          | **O(n)** | O(log n) |
| `remove`       | **O(n)** | O(log n) |
| `count`        | **O(n)** | O(log n) |
| `member`       | **O(n)** | O(log n) |
| `to_list`      | O(1) | O(n log n) |
| `size`         | O(n) | O(n) |

**Вывод:**  
- Реализация на **map** асимптотически быстрее для всех точечных операций
  (add, remove, count, member): **O(log n)** против **O(n)**.  
- Реализация на **списке** быстрее в `to_list`: **O(1)** против **O(n log n)**,
  так как внутренняя структура уже является списком пар.  
- При малом числе различных элементов (n ≤ 10–20) накладные расходы на map могут
  делать список практически быстрее из-за константных факторов, но при больших n
  map значительно выигрывает.

## Сложность `lab3_task1: merge(Tree1, Tree2)`

Используемый алгоритм: обойти Tree2 in-order (`to_list`), затем
вставить каждый элемент в Tree1 (`insert`).

- `to_list(Tree2)` — **O(m)**, где m — количество узлов в Tree2  
- Каждый `insert` в BST глубиной h — **O(h)**  
  - В лучшем случае (сбалансированное дерево): h = O(log(n + m))  
  - В худшем случае (вырожденное дерево → список): h = O(n + m)  
- Всего m вставок: **O(m · log(n+m))** среднее / **O(m · (n+m))** худшее  

Наша реализация — простой BST без балансировки (не AVL, не RB-tree),
поэтому **worst case: O(m·(n+m))** при вырожденных деревьях.

## Сложность `lab3_task3: union(Impl, MS1, MS2)`

- `to_list(MS1)` — O(n₁) [список] / O(n₁ log n₁) [map]  
- `to_list(MS2)` — O(n₂) [список] / O(n₂ log n₂) [map]  
- `add_all` для MS1: n₁ вызовов `add` → O(n₁²) [список] / O(n₁ log(n₁+n₂)) [map]  
- `add_all` для MS2: аналогично  

**Итоговая сложность union:**

| Реализация | Сложность union |
|------------|----------------|
| Список     | O((n₁ + n₂)²)  |
| Map        | O((n₁ + n₂) · log(n₁ + n₂)) |
